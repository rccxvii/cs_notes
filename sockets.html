üéØ Why Socket Pooling Matters

Creating a new socket is EXPENSIVE:
  • DNS lookup (find IP address)
  • TCP handshake (3-way communication)
  • SSL/TLS handshake (for HTTPS - can take 100ms+)

Without Connection Pooling (BAD ‚ùå):

Open socket ‚Üí Send request ‚Üí Close socket
Open socket ‚Üí Send request ‚Üí Close socket
Open socket ‚Üí Send request ‚Üí Close socket
‚ö†Ô∏è Slow and wasteful!

With Connection Pooling (GOOD ‚úÖ):

Open socket ‚Üí Send request 1 ‚Üí Send request 2 ‚Üí Send request 3 ‚Üí Eventually close
‚úÖ Fast and efficient - reuses existing connections!

üîí Socket Limits & The Bulkhead Connection

Your system has limits on how many sockets it can create:

File Descriptors

Each socket uses one

Linux: ~65,000 limit

Memory

Each socket consumes RAM

Can add up quickly!

Port Exhaustion

Client needs local ports

~64,000 ports available

üõ°Ô∏è This is why the Bulkhead Pattern is critical!

If one service exhausts all available sockets, other services can't make network calls. By limiting sockets per service (bulkheads), you prevent one bad service from bringing down everything.

new SocketsHttpHandler { MaxConnectionsPerServer = 15 // Max 15 sockets to this server } // This creates a pool of reusable sockets and prevents exhaustion!
